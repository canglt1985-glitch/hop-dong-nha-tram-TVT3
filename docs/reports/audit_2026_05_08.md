## 🔴 Lỗi nghiêm trọng (Phải sửa trước khi lên Production)

### 1. Hệ thống "Gọi dữ liệu sống" (N+1 External Request & Blocking)
- **Vị trí:** `backend/main.py` (Hàm `get_sites`)
- **Triệu chứng:** Mỗi khi mở web hoặc nhấn nút tải lại, backend sẽ gọi thẳng 3 request HTTP đồng bộ (`requests.get`) lên Google Sheets để tải hàng ngàn dòng dữ liệu và dùng thư viện `pandas` để parse CSV ngay lúc đó.
- **Hậu quả:** 
  - **Quá chậm:** Thời gian chờ có thể lên đến 5-10 giây cho một lần tải trang.
  - **Sập server:** Khi có nhiều người (toàn bộ team) cùng truy cập, hàm đồng bộ này sẽ "đóng băng" (block) toàn bộ luồng xử lý của server Python, khiến app bị sập hoặc báo lỗi Time Out.
  - **Bị khóa IP (Rate Limit):** Google Sheets sẽ khóa truy cập nếu bị spam quá nhiều request liên tục.
- **Phác đồ điều trị:** Áp dụng **Caching (Bộ nhớ đệm)**. Lần đầu tải dữ liệu sẽ lưu lại trên RAM/Redis khoảng 30-60 phút. Khi user vào, chỉ lấy từ Cache ra gửi về ngay lập tức (dưới 100ms). Chỉ khi nào user bấm nút "Đồng bộ" thì mới tải lại từ Google Sheets.

## 🟡 Cảnh báo (Nên sửa để trải nghiệm tốt hơn)

### 1. Thiếu cơ chế phân trang (Pagination) ở Frontend
- **Vị trí:** `frontend/src/App.tsx`
- **Triệu chứng:** Tải một lần toàn bộ danh sách hơn 1500 trạm về và ném thẳng vào thẻ `<div className="overflow-y-auto">`.
- **Hậu quả:** Trình duyệt của máy tính yếu sẽ bị đơ (lag) khi render hàng ngàn thẻ HTML cùng một lúc. Đặc biệt là nếu dùng trên điện thoại.
- **Phác đồ điều trị:** Dùng kỹ thuật Lazy Loading (chỉ render 20-50 trạm trên màn hình, cuộn chuột đến đâu render đến đó) hoặc phân trang đơn giản.

### 2. File cài đặt quá cồng kềnh
- **Vị trí:** Cấu trúc tải thư viện ở Frontend.
- **Triệu chứng:** Nhiều file assets, hình ảnh hoặc các logic tính toán chưa được tối ưu, làm tăng kích thước bộ cài (bundle size).
- **Phác đồ điều trị:** Setup React Lazy, Code Splitting ở Vite để trang web mở nhanh hơn ở lần đầu.
